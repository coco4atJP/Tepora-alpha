# Tepora 改善提案カタログ（2026-03-05）

**作成日**: 2026-03-05  
**対象**: Tepora v0.4.0（Rust Backend + React/Tauri Frontend）  
**目的**: 改善提案を一覧化し、「なぜ必要か」と「見込める効果」を明文化する。

---

## 1. 直近で効く改善（高ROI）

| ID | 提案 | なぜ必要か | 見込める効果 |
|---|---|---|---|
| ROI-01 | `infrastructure` を `episodic/knowledge/transport/observability` に再分割 | 現在の `infrastructure` は責務が広く、変更時の影響範囲が読みにくい | モジュール独立性が上がり、改修リードタイム短縮 |
| ROI-02 | `frontend/src/features` の依存方向を lint で強制 | Feature 間の逆依存・循環依存が増えると保守性が急落する | 依存崩壊を未然防止し、機能追加時の安全性向上 |
| ROI-03 | `secrets.yaml` の本運用化（keyring 連携＋暗号化保存＋ローテーション） | 秘密情報管理が段階運用のままだと、漏えい時の被害が大きい | セキュリティ強化、監査対応容易化 |
| ROI-04 | ルート `Taskfile.yml` と `Tepora-app/Taskfile.yml` の重複統合 | タスク定義が二重化すると運用差分・誤実行が発生する | 開発導線の一本化、手順の誤り削減 |
| ROI-05 | 起動時間・初回トークン時間・メモリ使用量 p95 を CI 記録 | 体感品質劣化がコードレビューだけでは検知しづらい | 性能退行の早期検知、品質ゲートの定量化 |
| ROI-06 | Actor キュー飽和時メトリクスを可視化 | `busy` 発生原因が不透明だと対処が場当たりになる | 輻輳原因の特定が速くなり、安定稼働率向上 |
| ROI-07 | 設定に `schema_version` と自動マイグレーター導入 | 設定項目追加時に互換性事故が起きやすい | 破壊的変更の回避、アップデート成功率向上 |
| ROI-08 | Wasm フィクスチャ成果物の掃除タスク追加 | テスト副産物の蓄積でリポジトリ・作業環境が汚れやすい | 開発環境健全化、CI の再現性向上 |
| ROI-09 | `legacy-peer-deps` 解除計画を依存更新 bot と自動追跡 | 一時回避策が恒久化すると将来の更新コストが膨らむ | 依存健全性の回復、将来負債の削減 |
| ROI-10 | `dev_sync` 動的ポート同期の E2E テスト追加 | 開発起動失敗は日次の生産性を直接落とす | 開発体験改善、セットアップ問い合わせ削減 |

## 7. 実行時の使い方（推奨）

1. 本カタログの ID（例: `ROI-05`, `SEC-03`）を使って課題管理に起票する。  
2. 各提案に対し、担当・期限・KPI（例: 起動時間 p95）を紐づける。  
3. 四半期ごとに「採用/保留/却下」を更新し、採用理由を記録する。

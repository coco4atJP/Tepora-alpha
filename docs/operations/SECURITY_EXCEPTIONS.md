# セキュリティ例外管理ポリシー

**最終更新**: 2026-02-26  
**対象バージョン**: v0.4.0

---

## 1. 目的

セキュリティ監査ツール（`cargo audit`, `npm audit`）で検出される脆弱性のうち、
プロジェクト固有の事情により即座に対処できないものを「例外」として管理する。
本ドキュメントは、例外の追加・維持・解除の判断基準と棚卸しルールを定める。

---

## 2. 例外管理台帳

### Rust（cargo audit）

| Advisory ID | パッケージ | 承認日 | 理由 | 次回レビュー |
|------------|-----------|-------|------|------------|
| RUSTSEC-2023-0071 | `rsa 0.9.x` | 2026-02-25 | `sqlx 0.8.6` の推移的依存（`sqlx-macros` → `sqlx-mysql` → `rsa`）。本プロジェクトは SQLite 専用で MySQL バックエンド・マクロ機能を使用しないため実質影響なし。`sqlx` から `rsa` 依存を除去する feature gate が提供されるまで除去不可 | 2026-06-01 |

### npm（npm audit）

現時点で ignore 設定なし。

---

## 3. 棚卸しルール

### 定期レビュー

- **頻度**: 四半期ごと（3月・6月・9月・12月）
- **担当**: セキュリティレビュー担当者
- **確認事項**:
  1. 上流パッケージで修正版がリリースされていないか
  2. 代替パッケージへの移行が可能になっていないか
  3. 影響範囲に変更がないか（新機能追加で利用するようになっていないか）

### 自動通知

`security-scan.yml` ワークフローが毎週月曜日に実行され、
新規脆弱性検出時に GitHub Issue を自動作成する。

---

## 4. 新規例外の追加基準

例外として ignore に追加するには、以下の条件を**すべて**満たす必要がある：

1. **影響評価**: プロジェクトの使用パターンにおいて脆弱性が実質影響しないことを確認
2. **除去不可の理由**: パッケージ除去・バージョン更新が不可能な技術的理由を記録
3. **文書化**: 本ドキュメントの台帳にエントリを追加（Advisory ID, 理由, 次回レビュー日）
4. **コード記録**: `audit.toml` と CI ワークフローの両方に、理由コメント付きで追加
